# TripKin 静态 Demo 统一整改范围

本文档记录本轮已经确认的团队执行范围，用于把现有静态页面统一成一个可演示的 TripKin 移动端 H5 Demo。

本整改计划以 `docs/tripkin-product-prd.md` 为产品依据，只记录本轮具体整改范围和验收标准。

本文档来源于本地审计整理和团队讨论提炼，不引用 `.local-docs/` 作为团队依据。

## 本轮目标

把现有页面统一成一条清楚的 Demo 主链路。当前公共路由和入口骨架已经完成，后续重点转为各页面按目录并行补齐内容和统一视觉。

```txt
Home -> MBTI 页面入口 -> MBTI -> 旅行身份卡 -> Map -> Bottle / Match -> Profile
```

本轮优先级：

1. 先确认公共骨架已经能挂载、能跳转、入口行为一致。
2. 各页面负责人按页面目录领取任务，尽量不互相修改对方页面。
3. 最后统一视觉和状态表达，让页面看起来像同一个产品。

`/map` 和 `/match` 暂作为结构和信息密度参考，但不要把它们的所有视觉细节硬套到其他页面。

## 不做范围

本轮仍以静态前端 Demo 为主，允许 `server/` 下的最小 mock API 链路用于 Match/Bottle 联调，但不代表真实后端能力完成。未配置 `VITE_API_BASE_URL` 时，页面继续走前端本地 mock。本轮仍不做：

- 数据库。
- 登录系统。
- 真实 AI。
- 真实聊天或好友系统。
- 真实图片上传。
- 真实匹配算法。
- 新 UI 组件库。
- 消息页。
- 外部短视频入口。
- 维度分析页。
- 深度推荐算法。
- 真实分享。

## 优先级

### P0：公共骨架验收和产品闭环

当前公共骨架必须满足：

- 首页 `/` 是常规入口。
- 底部导航为四栏：`首页 / 地图 / 匹配 / 我的`。
- Match 是底部主导航里的核心产品闭环入口。
- MBTI 是旅行人格生成 / 更新入口，不再作为底部导航 tab。
- Home、Profile 和必要页面内 CTA 可以承接 MBTI 引导或进入 `/mbti`。
- MBTI 模块目标流程为 `/mbti -> /mbti/test -> /mbti/result`。
- 不确定的远期能力不写入正式范围，先保留在飞书或 `.local-docs/`。

### P1：页面负责人并行补齐

公共骨架完成后，各页面负责人按目录领取任务：

- 首页负责人改 `src/pages/Home/`。
- MBTI 负责人改 `src/pages/Mbti/`。
- Map 负责人继续改 `src/pages/Map/`。
- Bottle 负责人继续改 `src/pages/Bottle/`。
- Match 负责人继续改 `src/pages/Match/`。
- Profile 负责人继续改 `src/pages/Profile/`。
- 需要改路由、`BottomNav`、`MbtiEntryModal` 时，先说明影响范围，不夹带单页重写。

### P2：页面并行补齐和统一

页面并行过程中继续处理：

- 统一字体、颜色、卡片、按钮、标签、弹层。
- 统一空状态、错误状态、toast 反馈。
- 优化地图搜索输入体验。
- 用人工 375px 走查回归跳转、截图和视觉统一问题。

## 模块整改要求

### MBTI

目标：作为 Demo 起点，生成可带入后续页面的旅行人格和身份卡。

必须完成：

- 目标路由拆分为 `/mbti`、`/mbti/test`、`/mbti/result`。
- `/mbti` 作为 MBTI 首页，介绍旅行 MBTI，并提供“开始测试”和“查看我的结果”入口。
- `/mbti/test` 作为测试页，复用现有题目和计分逻辑，不重写算法。
- `/mbti/result` 作为结果页，复用现有结果内容和旅行身份卡。
- 结果页增加“返回测试首页”入口。
- 每一题允许返回上一题重新选择。
- 测试完成后显示小人形象或形象化头像。
- 增加结果详情展示，说明当前旅行人格、标签、同行建议和适合目的地。
- 本轮不扩展题量，继续使用现有题库。

验收标准：

- 从 `/mbti` 可以进入 `/mbti/test`，完成后进入 `/mbti/result`。
- 结果页可以返回测试首页重新开始。
- 答题过程中可以返回上一题，重新选择后结果按新答案计算。
- 结果详情可打开或展开，375px 下文字不遮挡按钮。
- 身份卡上能看到 TripKin 旅行人格名称，例如“赛博特种兵”“精神卡皮巴拉”“精算系旅行家”“人文浪漫主义”。

### Home

目标：作为 Demo 常规入口，承接旅行探索、页面内 MBTI 引导和四栏底部导航。

必须完成：

- 新增 `/` 首页路由。
- 首页可展示旅行探索入口和进入地图的入口。
- 首页可触发 MBTI 引导弹窗。
- 已有 MBTI 结果时，可展示“我的旅行 MBTI”摘要入口。
- 页面内容先轻量落地，不在公共骨架阶段提前做完整首页运营内容。

验收标准：

- `/` 能正常打开。
- 首页底部导航包含首页、地图、匹配、我的。
- 点击首页的 MBTI 入口时，可以进入 MBTI 引导或 `/mbti`。
- 点击底部匹配入口时，可以进入 `/match`。
- 375px 下首页首屏没有横向滚动或关键按钮遮挡。

### Map

目标：作为 Demo 主枢纽，承接 MBTI 后的探索，并分发到 Bottle 和 Match。

必须完成：

- 地图图层中的漂流瓶入口能进入 `/bottle?dest=<目的地ID>`。
- “扔瓶子”入口能进入当前目的地的添加漂流瓶流程。
- 点击城市、区域或点位后，能进入对应目的地的漂流瓶内容。
- 地图区域卡片保留进入 `/match?dest=<目的地ID>` 的入口。
- 无真实地图 key 时继续使用静态地图兜底，演示时不被错误提示遮挡主要按钮。

验收标准：

- 从地图选中任一已有 region/spot 后，可以进入对应 `/bottle`。
- Bottle 页面能显示同一个目的地。
- 从同一地图卡片进入 Match 后，Match 页面也能显示同一个目的地。
- 图层、搜索、选区卡片和主要 CTA 在 375px 下可点击。

### Bottle

目标：作为目的地内容分支，展示对应目的地的旅行故事、心愿和漂流瓶互动。

必须完成：

- 按 `src/pages/Map/data/mapData.ts` 里已有 region/spot 准备独立 mock 漂流瓶数据。
- 根据 URL query 或共享状态读取当前目的地。
- 列表只展示当前目的地相关的漂流瓶内容。
- 点击列表卡片打开详情弹层。
- 添加漂流瓶使用弹层或底部抽屉，提交后更新当前目的地列表。
- 提供到 Match 的入口，例如“找同目的地搭子”。

验收标准：

- `/bottle?dest=<目的地ID>` 能显示对应目的地名称。
- 不同目的地看到的漂流瓶列表不同。
- 点击漂流瓶卡片能打开详情，关闭后回到列表。
- 添加漂流瓶成功后，新内容出现在当前目的地列表。
- 点击 Match 入口进入 `/match?dest=<同一目的地ID>`。

### Match

目标：作为目的地社交分支，展示同目的地搭子和行程，并结合 TripKin 旅行人格。

必须完成：

- 页面读取当前目的地，替换固定目的地展示。
- 卡片除展示 16 型 MBTI 字母外，也展示 TripKin 旅行人格结果。
- TripKin 旅行人格以 `personaId` 为准，不用 `mbtiTypeEn` 当作 16 型 MBTI 字母。
- 保留筛选弹层、他人身份卡弹层、加入行程弹层。
- 处理筛选 chip 在 375px 下的截断和可触达问题。

验收标准：

- `/match?dest=<目的地ID>` 能显示对应目的地。
- 卡片上同时能看到 MBTI 字母和 TripKin 旅行人格标签。
- 筛选、查看 TA、加入行程都能打开对应弹层并关闭。
- 375px 下筛选 chip 不造成不可理解的裁切。

### Profile

目标：从设置占位页改成“我的旅行身份”页面，承接 MBTI 和后续互动。

必须完成：

- 首屏展示用户旅行身份卡。
- 展示 MBTI/TripKin 旅行人格结果、旅行标签、目的地倾向。
- 展示漂流瓶、搭子匹配、旅行记忆等摘要入口。
- 设置入口后置，不作为首屏主体。

验收标准：

- `/profile` 首屏能看出用户是谁、是什么旅行人格、适合怎样旅行。
- 页面不再以 `--` 占位作为主要内容。
- 375px 下身份卡和摘要入口不重叠、不横向滚动。

### Cross-page

目标：让页面像同一个 TripKin 产品，而不是多个独立应用拼接。

必须完成：

- 全局入口关系统一：底部导航固定为 `首页 / 地图 / 匹配 / 我的`。
- MBTI 入口由 Home、Profile 和必要页面内 CTA 承接；不再由 BottomNav 承接。
- 公共路由骨架已完成，后续让各页面负责人并行开发。
- 统一字体层级、主色、辅助色、背景逻辑。
- 统一卡片圆角、阴影、内边距和信息密度。
- 统一主按钮、次按钮、标签、toast、底部弹层。
- 统一返回逻辑：Map 是主枢纽，Bottle/Match 返回 Map。
- 页面私有组件先留在页面目录；只有确认跨页复用后，再抽到 `src/components`。

验收标准：

- `/`、`/mbti`、`/map`、`/bottle`、`/match`、`/profile` 截图并排看，不像不同项目拼在一起。
- 375px 下没有横向滚动或关键内容遮挡。
- 每个主页面都有清楚的下一步入口。

## 公共骨架完成后的协作方式

当前公共骨架已经包含：

1. 首页路由 `/`。
2. `/mbti/test`、`/mbti/result` 路由入口。
3. `首页 / 地图 / 匹配 / 我的` 四栏 `BottomNav`。
4. `MbtiEntryModal` 页面内 MBTI 引导弹窗，可被明确的 MBTI CTA 使用。

后续不建议多人同时修改路由、底部导航和全局入口。各页面负责人应按 `src/pages/<PageName>/` 分目录推进；确实需要修改公共骨架时，先说明影响范围。

## 公共约定

### 目的地上下文

首轮目的地上下文使用现有 Zustand store 或 URL query 承接。

推荐路由格式：

```txt
/bottle?dest=<目的地ID>
/match?dest=<目的地ID>
```

`<目的地ID>` 优先使用 `src/pages/Map/data/mapData.ts` 中已有的 region/spot `id`。

首轮只覆盖地图已有 region/spot，不额外扩展全量城市库。

### 旅行人格

TripKin 旅行人格使用 `personaId` 作为稳定标识：

```txt
Cyber-Raider
Zen-Capybara
Budget-Architect
Romantic-Observer
```

展示文案可使用中文名称：

```txt
赛博特种兵
精神卡皮巴拉
精算系旅行家
人文浪漫主义
```

注意：

- `mbtiTypeEn` 当前是旅行人格英文标题，不是 16 型 MBTI 字母。
- Match 卡片里的 `mbti` 字段才是 16 型 MBTI 字母，例如 `ENFP`、`INTJ`。
- 后续做匹配展示时，不要把 `mbtiTypeEn` 直接当作 Match 的 `mbti` 使用。

## 协作顺序

1. 先确认公共骨架能正常运行。
2. 各页面负责人完成自己负责范围内的页面内容。
3. 再做跨页面跳转、目的地上下文和返回逻辑联调。
4. 最后统一视觉样式和可复用组件。
5. 不先做大规模重构，不先抽象不确定复用的组件。
6. 需要改路由、`BottomNav`、全局入口组件时，先说明影响范围，不夹带单页视觉重写。

多人协作时，优先保持页面内改动就近落在对应 `src/pages/<PageName>/` 目录。涉及共享状态、路由、全局样式或公共组件时，需要先说明影响范围。

## 验收方式

每轮改完至少检查：

```bash
npm run lint
npm run build
```

如果改了 Markdown 或格式相关内容，也检查：

```bash
npm run format:check
```

需要验证页面跳转、截图或控制台错误时，团队默认使用人工 375px 走查、Chrome DevTools 或普通浏览器检查。

`webapp-testing` 只作为少数本地环境已具备该能力的成员的可选辅助方式；其他成员不一定有这套能力，AI 或协作者不要主动要求团队成员安装或启用它。只有任务明确点名 `webapp-testing`、Playwright、自动化验证或截图验证，并且当前环境确实可用时（可以提问一次是否主动安装），才按 `docs/webapp-testing-guide.md` 执行。临时脚本、截图和日志仍放在 `.venv/`，不提交到正式目录。
