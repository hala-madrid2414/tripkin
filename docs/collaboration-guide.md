# 协作说明

本文档说明 TripKin 当前阶段的团队协作方式。目标是让每个人能按页面推进，减少互相覆盖、冲突和重复沟通。

## 当前协作方式

当前按页面或目录分工开发，每个人优先负责自己范围内的内容。

页面负责人、目录归属和具体需求以飞书或当次任务说明为准。仓库文档不维护未确认的分工表。

## 开发前

新拉代码后，先安装依赖：

```bash
npm install
```

每次开始开发前：

1. 拉最新代码。
2. 确认自己要改哪个页面或目录。
3. 运行 `npm run dev`，确认项目能打开。
4. 不确定代码放哪里时，先看 `docs/coding-guide.md`。

如果项目启动失败并提示缺少依赖，优先确认是否忘记执行 `npm install`。

## 开发中

必须：

- 尽量只改自己负责的页面或目录。
- 不随便改别人页面的主结构。
- 不确定需求先写到飞书讨论，不直接写死到仓库。
- 新增依赖、组件库、后端、真实 AI 接口前必须先确认。

推荐：

- 页面内部内容先就近写。
- 重复 UI 出现后，再考虑抽到 `src/components`。
- 页面内部大功能块可以抽到 `src/modules`。
- 普通页面内部变更不需要写决策记录。

公共部分处理原则：

- 已有确认可用的公共样式 / 公共组件，优先复用。
- 只服务自己页面的内容，优先写在自己负责的页面目录里。
- 页面内容变复杂时，优先在自己页面目录内拆组件，并为复杂组件配套 `index.tsx + ComponentName.module.less`。
- 确认多个页面都会复用后，再讨论是否抽到 `src/components`。
- 不要随意修改已经确定的公共组件或全局样式。
- 不要未经确认引入组件库。
- 不要把页面私有逻辑放进公共目录。
- 涉及公共部分变更时，先说明原因和影响范围；影响多人协作的，同步写决策记录。

## 本地开发资料与草稿区

团队成员可以使用 `.local-docs/` 保存个人开发草稿、AI 对话总结、截图参考、分步计划和未确认需求拆解。复杂页面、参考图较多或需求尚未完全收敛时，建议先在 `.local-docs/` 建立本地上下文，再提炼到正式文档。

`.local-docs/` 不进入 Git 提交，不作为团队正式需求、规范或决策依据。

如果本地已经确认了一些只对当前任务有效、但不适合进入团队文档的规则，可以写到 `.local-docs/rules.md`。这个文件只作为本地上下文，不能覆盖 `AGENTS.md`、`docs/` 中的正式规则，也不应被直接同步进仓库文档。

当本地草稿中的内容已经确认会影响多人协作、公共规则或后续开发理解时，需要提炼后同步到正式 `docs/`，必要时记录到 `docs/decision-notes/README.md`。

如果 AI 或协作者发现 `.local-docs/` 中有参考图，不能直接猜测使用方式，必须先确认：

- 是严格还原，还是只做风格参考？
- 如果是参考，具体参考布局、色彩、组件层级、信息密度、交互状态、文案还是素材？
- 哪些部分不能改，哪些部分可以调整？

推荐提问模板：

```txt
我看到 .local-docs 里有参考图。请确认这次使用方式：

1. 严格还原：布局、尺寸、颜色、层级和交互状态都尽量贴近参考图。
2. 风格参考：只参考整体气质，具体布局和组件可以按项目现状调整。
3. 指定参考：只参考你指定的部分，例如配色 / 卡片 / 弹层 / 筛选区。
```

## 什么时候写决策记录

如果一个改动会影响多人协作或后续开发理解，需要记录到 `docs/decision-notes/README.md`。

需要记录：

- 新增或修改全局样式变量
- 新增依赖
- 引入组件库
- 修改路由结构
- 修改目录结构
- 修改工程化配置
- 抽跨页面公共组件

不需要记录：

- 页面内部样式
- 页面内部 mock 数据
- 页面内部小组件
- 文案调整
- 修自己页面里的 bug

## 提交前

提交前至少运行：

```bash
npm run lint
npm run build
```

如果改了 Markdown 或格式相关内容，可以额外运行：

```bash
npm run format:check
```

## 本地页面验证

当前项目没有引入正式测试框架。提交前必须检查仍然是 `npm run lint` 和 `npm run build`。

这是赶时间的前端 Demo 项目，`webapp-testing` 不是必需项，也不是默认开发流程。只有当用户或任务明确提到测试、自动化验证、截图验证、Playwright、`webapp-testing` 或需要可重复验证时，AI 或协作者才使用它。

如果需要可重复地验证本地页面、交互、控制台报错或截图，可以按 `docs/webapp-testing-guide.md` 中的流程执行。它只作为本地辅助验证方式，不替代人工视觉走查，也不代表产品最终验收。

首次使用这套流程时，需要先引导创建本地 `.venv/`。临时 Playwright 脚本、截图、trace、log 和其他运行产物都放在 `.venv/` 内；人工整理后的测试结论或问题记录可以放在 `.local-docs/`。`.venv/` 不进入 Git；不使用这套流程的团队成员不需要创建它。

不要把某个人本机 AI 目录里的 `webapp-testing` skill 复制进仓库。如果当前 AI 环境没有安装这项能力，先说明缺失情况，并引导使用者在自己的 AI 工具环境中安装或启用；不同机器可能不是 `.codex` 目录。临时无法安装时，退回 Codex 自带浏览器或人工检查。

## 提交信息

推荐使用：

```bash
npm run commit
```

提交格式只使用：

```txt
feat: 添加页面内容
feat(map): 添加地图页面内容
```

推荐正文写法：

```txt
feat(map): 添加地图页面静态内容

- 添加城市点位展示
- 添加底部目的地卡片
```

不要使用：

```txt
feat(): 添加页面内容
update
改了一下
```

## 分支建议

仓库里只写最小规则，详细 Git 教程放飞书。

建议分支名：

```txt
name/feat-map-page
name/fix-mbti-style
```

如果不确定如何处理分支或冲突，不要自己乱合并，先截图发群里找负责人。

## 遇到问题

页面跑不起来：

- 先确认是否执行过 `npm install`。
- 再截图终端报错。
- 不要只说“跑不了”，要带上命令和报错。

合并冲突：

- 不要直接点“接受全部”。
- 先看冲突文件是不是别人负责的页面。
- 不确定时找负责人一起处理。

需求不确定：

- 写到飞书讨论。
- 不要把未确认内容写进仓库文档或代码。
