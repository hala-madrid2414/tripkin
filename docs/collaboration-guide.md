# 协作说明

本文档说明 TripKin 当前阶段的团队协作方式。目标是让每个人能按页面推进，减少互相覆盖、冲突和重复沟通。

## 当前协作方式

当前按页面分工开发，每个人优先负责自己页面内的内容。

主流程页面：

- `/mbti`
- `/map`
- `/bottle`
- `/match`

页面负责人和具体需求以飞书为准。仓库文档不维护未确认的分工表。

## 开发前

新拉代码后，先安装依赖：

```bash
npm install
```

每次开始开发前：

1. 拉最新代码。
2. 确认自己要改哪个页面或目录。
3. 运行 `npm run dev`，确认项目能打开。
4. 不确定代码放哪里时，先看 `docs/coding-guide.md`。

如果项目启动失败并提示缺少依赖，优先确认是否忘记执行 `npm install`。

## 开发中

必须：

- 尽量只改自己负责的页面或目录。
- 不随便改别人页面的主结构。
- 不确定需求先写到飞书讨论，不直接写死到仓库。
- 新增依赖、组件库、后端、真实 AI 接口前必须先确认。

推荐：

- 页面内部内容先就近写。
- 重复 UI 出现后，再考虑抽到 `src/components`。
- 页面内部大功能块可以抽到 `src/modules`。
- 普通页面内部变更不需要写决策记录。

## 什么时候写决策记录

如果一个改动会影响多人协作或后续开发理解，需要记录到 `docs/decision-notes/README.md`。

需要记录：

- 新增或修改全局样式变量
- 新增依赖
- 引入组件库
- 修改路由结构
- 修改目录结构
- 修改工程化配置
- 抽跨页面公共组件

不需要记录：

- 页面内部样式
- 页面内部 mock 数据
- 页面内部小组件
- 文案调整
- 修自己页面里的 bug

## 提交前

提交前至少运行：

```bash
npm run lint
npm run build
```

如果改了 Markdown 或格式相关内容，可以额外运行：

```bash
npm run format:check
```

## 提交信息

推荐使用：

```bash
npm run commit
```

提交格式只使用：

```txt
feat: 添加页面内容
feat(map): 添加地图页面内容
```

推荐正文写法：

```txt
feat(map): 添加地图页面静态内容

- 添加城市点位展示
- 添加底部目的地卡片
```

不要使用：

```txt
feat(): 添加页面内容
update
改了一下
```

## 分支建议

仓库里只写最小规则，详细 Git 教程放飞书。

建议分支名：

```txt
name/feat-map-page
name/fix-mbti-style
```

如果不确定如何处理分支或冲突，不要自己乱合并，先截图发群里找负责人。

## 遇到问题

页面跑不起来：

- 先确认是否执行过 `npm install`。
- 再截图终端报错。
- 不要只说“跑不了”，要带上命令和报错。

合并冲突：

- 不要直接点“接受全部”。
- 先看冲突文件是不是别人负责的页面。
- 不确定时找负责人一起处理。

需求不确定：

- 写到飞书讨论。
- 不要把未确认内容写进仓库文档或代码。
