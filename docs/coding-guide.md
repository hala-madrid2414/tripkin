# 代码位置与样式规范

本规范用于 TripKin **MVP 闭环奠基期**。目标是让多人和 AI 能按阶段推进产品闭环，减少目录混乱、样式冲突、假交互和一次性过度开发。

仓库文档只记录当前已经确认的规则。未确认的产品想法、远期规划和技术方案，不写成团队规则。

## 阶段原则

当前阶段优先补齐产品闭环，而不是继续堆页面展示。

实施顺序：

1. 前端闭环：用 Zustand/local storage 和本地阶段性数据源打通用户资产。
2. 后端闭环：将用户资产和行为迁移到 `server/` API。
3. AI 闭环：引入结构化旅行卡，并让卡片进入保存、投瓶、找搭子和 Profile 沉淀。

未进入当前阶段的能力不要擅自加入，包括数据库、完整登录、真实 AI、新 UI 组件库、复杂 CI 和正式测试框架。

## 目录职责

### `src/pages`

放带路由的大页面。

页面目录通常按路由页面或业务页面命名：

```txt
PageName/
  index.tsx
  PageName.module.less
```

页面内部小组件优先留在对应页面目录内。只有已经确认跨页面复用，才放进 `src/components`。

### `src/modules`

放页面内部的大功能块。模块不单独挂路由，通常服务某个页面或一段页面流程。

如果只是页面内部的小片段，先就近放在页面目录内，不提前抽模块。

### `src/layouts`

放应用级页面外壳，例如 `src/layouts/AppLayout/`。

Layout 负责跨路由页面框架规则，例如是否显示底部导航、主 Tab 安全区、页面壳宽度和导航上下文。

只有 layout 能决定 `BottomNav` 是否出现；页面不直接 `import BottomNav`。

底部主导航固定为：

```txt
首页 / 地图 / 匹配 / 我的
```

主 Tab 白名单：

```txt
/ /map /match /profile
```

`/mbti`、`/mbti/test`、`/mbti/result` 和 `/bottle` 是流程页或内容分支，不显示底部导航。

### `src/components`

放跨页面复用 UI。

不要因为“可能会复用”就提前抽公共组件。只服务单个页面的内容，优先放在对应页面目录内。

已确认的公共组件不要为了单页需求随意修改。确实需要新增或修改公共组件时，先说明影响范围；如果会影响多人协作，同步记录到 `docs/decision-notes/README.md`。

### `src/services`

放请求和数据访问逻辑。

`src/services` 是页面访问数据的门面，可以在本地阶段性数据源和 `server/` API 之间切换。页面不关心数据来自本地、内存 API 还是后续正式服务。

规则：

- 页面不要直接拼后端 URL。
- Match/Bottle 已支持 `VITE_API_BASE_URL` 双通道。
- 后续用户资产、保存、申请、邀请和 AI 卡片也应优先通过 service 收口。
- 如果某个动作尚未进入阶段，service/API 设计应先写入计划或审计文档，不在页面里假装完成。

### `server`

放 TypeScript Express 后端代码。

`server/` 是 MVP API 雏形，当前以内存和阶段性数据源承载。它用于逐步承接前端闭环中的用户资料、用户资产、行为记录和后续 AI 卡片接口。

规则：

- 路由、后端数据、后端类型和服务逻辑都放在 `server/src` 下。
- 不直接引用前端页面目录里的 UI 数据文件。
- 新增 API 前先确认前端闭环需求和接口草案。
- 当前不要擅自加入数据库、完整登录、真实 AI、真实匹配算法、复杂 CI 或正式测试框架。

### `src/store`

放跨页面共享的 Zustand 状态。

适合放 store：

- 用户 MBTI 结果。
- 当前选择的目的地。
- 用户资产模型，包括保存内容、投出的漂流瓶、收藏的漂流瓶、收藏的目的地、收藏的搭子、申请加入的行程、发起的同行邀请、最近 AI 卡片、我的行程、我的足迹、我的成就。
- 跨页面需要读取的匹配上下文。

不适合放 store：

- 按钮 hover。
- 弹窗开关。
- 当前页面内部 tab。
- 一次性的页面内部表单状态。

### `src/styles`

放全局样式、重置样式和全局变量。

不要随意新增全局 class。页面和组件样式默认使用 CSS Modules。

### `src/types`

放跨页面或跨模块复用的 TypeScript 类型。页面私有类型可以先放在页面内部。

### `src/utils`

只放纯工具函数。

不要放：

- JSX。
- React Hooks。
- 页面业务逻辑。
- 请求逻辑。

## 用户行为闭环规则

涉及提交、保存、收藏、点赞、关注、申请、邀请、编辑资料等用户行为时，不能只显示成功 Toast。

必须满足以下之一：

- 写入 `src/store` 中的共享状态。
- 通过 `src/services` 写入本地阶段性数据源。
- 通过 `src/services` 写入 `server/` API。
- 如果当前阶段尚未实现写入路径，UI 文案必须明确是占位状态。

不要新增页面内孤立 mock 列表来伪装行为结果。凡是 Profile 需要展示的内容，都应优先进入用户资产模型。

## 命名规则

必须：

- 文件夹、组件名、路由 path 使用英文。
- 页面展示文案使用中文。
- 页面和组件样式使用 `*.module.less`。
- 入口组件文件使用 `index.tsx`。

推荐：

- 组件文件夹使用 PascalCase。
- 样式文件使用组件名，例如 `IdentityCard.module.less`。

## 移动端尺寸基准

当前按移动端 H5 开发，视觉和验收基准宽度为 `375px`。

必须：

- 页面宽度使用 `width: 100%` 或自然块级宽度。
- 页面最小高度使用 `min-height: 100svh`。
- 页面内容允许自然向下滚动。
- 不把整页写死成 `width: 375px`。
- 不把整页写死成固定高度画布。
- 避免横向滚动。

当前阶段不做复杂响应式系统，不引入 `rem`、`vw` 全局缩放或额外适配库。

## 单位规则

当前可以直接使用 `px`。

适合使用 `px`：

- 字号。
- 间距。
- 圆角。
- 图标尺寸。
- 卡片内边距。
- 按钮高度。

需要保持弹性：

- 页面容器宽度。
- 长文案。
- 列表。
- 地图区。
- 底部操作区。
- 弹窗宽度。

## 全局样式变量

新增或修改全局变量前，需要确认它会被多个页面复用。

如果变更影响多人协作，需要记录到 `docs/decision-notes/README.md`。

当前 TripKin 视觉 token 位于 `src/styles/variables.less`。页面优先使用这些变量，不要在页面里重复散写主色、卡片阴影、圆角和文字颜色。

## 组件库

当前已确认可以在需要时使用 `antd-mobile` 加速移动端基础交互。

使用时遵守 `docs/antd-mobile-usage-guide.md`：优先用它处理基础交互，不让默认组件视觉决定页面风格。

除 `antd-mobile` 外，未经确认不要引入其他组件库。

## 当前阶段立即执行项

1. 每次改动先明确属于前端闭环、后端闭环还是 AI 闭环。
2. 页面必须在 375px 宽度下正常展示。
3. 涉及用户行为成功反馈时，必须说明写入 store/service/API 的位置。
4. 不确定的需求写到计划或审计文档，不直接写进代码。
5. 新增依赖、数据库、完整登录、真实 AI、新 UI 组件库、复杂 CI 或正式测试框架前必须先确认。
6. 源码变更提交前运行 `npm run lint` 和 `npm run build`；纯文档变更可只按需运行 `npm run format:check`。
