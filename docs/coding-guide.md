# 代码位置与样式规范

本规范用于 TripKin 当前前端 Demo 阶段，目标是让多人按页面协作开发时，减少目录混乱、样式冲突和 AI 随意改代码的问题。

仓库文档只记录当前已确认的开发规则。未确认的产品设想、远期规划和技术方案，不写成仓库规则。

## 规范强度

- 必须：所有人和 AI 都要遵守。
- 推荐：优先遵守，特殊情况可以说明原因。
- 暂未确定：不要自行引入或写入项目。

## 目录职责

### `src/pages`

放带路由的大页面。页面目录通常按路由页面或业务页面命名。

页面目录使用：

```txt
PageName/
  index.tsx
  PageName.module.less
```

如果页面内部已经拆出复杂组件，可以继续放在当前页面目录内，例如：

```txt
PageName/
  index.tsx
  PageName.module.less
  components/
    FeatureBlock/
      index.tsx
      FeatureBlock.module.less
```

这类页面私有组件不等于跨页面公共组件，不需要因为拆了目录就放进 `src/components`。
页面根 `PageName.module.less` 只负责页面壳、整体布局和少量页面级变量；复杂组件的视觉细节优先写进自己的 `ComponentName.module.less`。

### `src/modules`

放页面内部的大功能块。模块不单独挂路由，通常服务于某个页面或某段页面流程。

如果只是页面内部的小片段，先就近写在页面目录内，不需要提前抽模块。

### `src/components`

放跨页面复用的 UI 组件。

如果项目中已经存在确认可用的公共组件或公共样式，开发页面时应优先复用，不要重复实现一份相同能力。

当前阶段不要因为“可能会复用”就提前抽公共组件。只服务单个页面的内容，优先放在对应页面目录内；只有已经确认复用，或明显会被多个页面使用时，再放到 `src/components`。

已确认的公共组件不要为单个页面需求随意修改。确实需要新增或修改公共组件时，先说明原因和影响范围；如果变更会影响多人协作，需要记录到 `docs/decision-notes/README.md`。

### `src/services`

放前端请求和数据访问逻辑。当前阶段可以先作为本地 mock 数据门面，不要为了完整硬写真实接口。

后续出现真实请求时，页面里不要散写请求逻辑，应通过 `services` 收口。前端页面不要直接拼后端 URL。

### `server`

放最小 TypeScript Express 后端代码，用于 mock API 和后续前后端联调。

后端目录使用独立 `package.json` 和 `tsconfig.json`。路由、后端 mock 数据、后端类型和服务逻辑都放在 `server/src` 下，不要直接引用前端页面目录里的 UI mock 文件。

当前 `server/` 只承接已确认的最小 mock API；不要自行加入数据库、登录、真实 AI、真实匹配算法、复杂 CI 或正式测试框架。

### `src/store`

放跨页面共享的 Zustand 状态。

适合放 store：

- 用户 MBTI 结果
- 当前选择的目的地
- 跨页面需要使用的匹配信息

不适合放 store：

- 按钮 hover
- 弹窗开关
- 当前页面内部 tab
- 一次性的页面内表单状态

### `src/styles`

放全局样式、重置样式和全局变量。

不要随意新增全局 class。页面和组件样式默认使用 CSS Modules。

### `src/types`

放跨页面或跨模块复用的 TypeScript 类型。

页面私有类型可以先放在页面内部，不需要提前抽到 `types`。

### `src/utils`

只放纯工具函数。

不要放：

- JSX
- React Hooks
- 页面业务逻辑
- 请求逻辑

## 命名规则

必须：

- 文件夹、组件名、路由 path 使用英文。
- 页面展示文案使用中文。
- 页面和组件样式使用 `*.module.less`。
- 入口组件文件使用 `index.tsx`。

推荐：

- 组件文件夹使用 PascalCase，例如 `IdentityCard`。
- 样式文件使用组件名，例如 `IdentityCard.module.less`。

示例：

```txt
正确：
src/pages/Map/
  index.tsx
  Map.module.less

不推荐：
src/pages/map.tsx
src/pages/Map/index.less
```

## 移动端尺寸基准

当前项目按移动端 H5 开发，视觉和开发基准宽度为 `375px`。

开发时优先使用 Chrome DevTools 的 iPhone SE / 375px 宽度检查页面。

当前阶段不做复杂响应式系统，不引入 `rem`、`vw` 全局缩放或额外适配库。

## 布局规则

必须：

- 页面宽度使用 `width: 100%` 或自然块级宽度。
- 页面最小高度使用 `min-height: 100svh`。
- 页面内容允许自然向下滚动。
- 不把整页写死成 `width: 375px`。
- 不把整页写死成固定高度画布。
- 避免横向滚动。

推荐：

- 主要布局优先使用 `flex` / `grid`。
- 桌面预览时让移动端页面居中或保持合理最大宽度。

## 单位规则

当前阶段可以直接使用 `px`。

适合使用 `px` 的内容：

- 字号
- 间距
- 圆角
- 图标尺寸
- 卡片内边距
- 按钮高度

需要保持弹性的内容：

- 页面容器宽度
- 长文案
- 列表
- 地图区域
- 底部操作区
- 弹窗宽度

一句话规则：`px` 可以用，固定画布不可以。

## 全局样式变量

新增或修改全局变量前，需要确认它会被多个页面复用。

如果变更会影响多人协作，需要记录到 `docs/decision-notes/README.md`。

记录时说明：

- 变量名
- 变量值
- 使用场景
- 为什么不能使用现有变量
- 影响哪些页面

当前 TripKin 视觉 token 位于 `src/styles/variables.less`。后续正式页面样式优先使用这些变量，不要在页面里重复散写一套主色、卡片阴影、圆角和文字颜色。

`.local-docs/` 仍是本地草稿区，不直接作为正式需求文档提交；当其中的视觉结论会影响多人协作时，需要提炼后写入正式 `docs/` 或在本地执行计划中明确边界。

当 `.local-docs/` 中的参考图已经被确认是高保真目标时，正式文档或本地执行计划必须写明还原范围、不可偏离项和可以自由调整的部分。确认后的高保真参考图优先级高于旧草稿里的“氛围参考”描述。

旧占位页不是视觉参考对象。不要按占位页颜色推导新页面风格，也不要为了“统一”重写无关页面。只需要保证全局变量调整后现有页面仍能正常构建和基础展示。

## 组件库

当前阶段已确认可以在需要时使用 `antd-mobile` 加速移动端 Demo 开发。

使用时遵守 `docs/antd-mobile-usage-guide.md`：优先用它处理基础交互，不直接让默认组件视觉决定页面风格。

除 `antd-mobile` 外，未经团队确认，不要自行安装或引入其他组件库。

## 当前阶段立即执行项

1. 每个人优先改自己负责的页面。
2. 页面必须能在 375px 宽度下正常展示。
3. 提交前运行 `npm run lint` 和 `npm run build`。
4. 不确定的需求写到飞书，不写进仓库代码或文档。
5. 新增依赖、组件库、后端能力、AI 接口前必须先确认。
