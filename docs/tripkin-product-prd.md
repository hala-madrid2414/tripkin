# TripKin 产品 PRD

本文档是 TripKin 当前 **MVP 闭环奠基期** 的产品说明。它记录已经确认的产品方向、阶段边界、页面职责和最小验收标准。

TripKin 不是一次性完整商业化重写项目。当前目标是按阶段把移动 H5 旅行陪伴产品从页面型演示推进到上线产品雏形：先打通前端用户资产闭环，再迁移到后端 API，最后引入结构化 AI 旅行卡。

## 产品定位

TripKin 是一个移动端 H5 旅行搭子与旅行陪伴产品，服务“旅行前不知道去哪、和谁去、怎么玩”的轻量决策场景。

当前产品主链路升级为：

```txt
Home / Map / AI 灵感 -> Bottle / Match 行为 -> UserAsset -> Profile 沉淀
```

其中：

- Home 负责常规入口、搜索、推荐和后续 AI 灵感入口。
- MBTI 负责生成旅行人格。
- Map 负责目的地探索和目的地上下文。
- Bottle 负责目的地内容互动和投瓶。
- Match 负责搭子 / 行程匹配、申请和邀请。
- Profile 负责用户身份与资产沉淀。

## 当前阶段

当前阶段为 **MVP 闭环奠基期**。

阶段 1：前端闭环

- 使用 Zustand/local storage 和本地阶段性数据源打通用户资产。
- 用户行为必须产生可追踪结果，包括投瓶、收藏、关注、申请、邀请、资料编辑。
- Profile 读取同一份用户资产，而不是只展示孤立页面数据。

阶段 2：后端闭环

- `server/` 承接用户资料、用户资产和行为记录 API。
- 前端继续通过 `src/services` 访问数据。
- 后端当前以内存和阶段性数据源承载，不直接引入数据库。

阶段 3：AI 闭环

- AI 输出结构化旅行卡。
- 旅行卡可以保存、投瓶、找搭子、申请行程。
- Profile 展示用户最近生成、浏览、保存和使用过的 AI 卡片。

## 页面职责

### Home

Home 是产品入口，负责展示旅行探索入口、搜索、推荐和 MBTI 入口。后续 AI 灵感入口也优先从 Home 或 Map 承接。

必须包含：

- 旅行探索入口。
- 进入 Map 的入口。
- 触发 MBTI 引导或进入 MBTI 的页面内入口。
- 已有 MBTI 结果摘要或进入 MBTI 的提示。
- 进入 Bottle 和 Match 的明确入口。

行为闭环要求：

- 搜索或推荐如果指向具体目的地，应带 `dest` 或写入当前目的地。
- 后续 AI 灵感卡保存后应进入用户资产。

### MBTI

MBTI 负责完成测试并生成用户旅行人格。

必须包含：

- `/mbti` 首页。
- `/mbti/test` 测试流程。
- `/mbti/result` 测试结果。
- 旅行身份卡。
- 进入地图的入口。

行为闭环要求：

- 结果写入 `useTripStore`，并在刷新后保留。
- Profile 应读取同一份旅行人格数据。

### Map

Map 是目的地探索枢纽，负责承接 MBTI 后的目的地探索，并分发到 Bottle 和 Match。

必须包含：

- 地图或静态地图兜底。
- 目的地搜索 / 筛选。
- 区域、城市或点位信息。
- 当前目的地上下文。
- 进入漂流瓶和搭子匹配的入口。

行为闭环要求：

- 选择目的地应更新当前目的地。
- 进入 Bottle / Match 时应带同一目的地。
- 后续收藏目的地、浏览足迹应进入用户资产。

### Bottle

Bottle 是目的地内容互动分支，负责展示当前目的地的旅行故事、心愿和漂流瓶互动。

必须包含：

- 当前目的地名称。
- 当前目的地的漂流瓶列表。
- 漂流瓶详情。
- 添加漂流瓶。
- 进入同目的地 Match 的入口。

行为闭环要求：

- 投瓶写入用户资产和当前列表。
- 点赞、收藏、关注不能只改当前页面 state；阶段 1 应写入用户资产。
- 打招呼如果尚未进入阶段，应明确是占位状态，不应假装完成。

### Match

Match 是目的地社交分支，负责展示同目的地搭子和行程。

必须包含：

- 当前目的地名称。
- 搭子 / 行程卡片。
- 筛选。
- 他人身份卡。
- 加入行程。
- MBTI 字母和 TripKin 旅行人格展示。

行为闭环要求：

- 申请加入行程应写入 `TripApplication`。
- 发起同行邀请应写入 `CompanionInvitation`。
- 筛选如果只是占位，必须明确说明。

### Profile

Profile 是用户身份和资产沉淀页，不是静态设置页。

必须包含：

- 用户旅行身份卡。
- MBTI / TripKin 旅行人格。
- 旅行标签。
- 目的地倾向。
- 漂流瓶、搭子匹配、旅行记录、收藏、行程、足迹和成就摘要。

行为闭环要求：

- 资料编辑必须写入统一资料源。
- 收藏、投瓶、申请、邀请、AI 卡片应从用户资产读取。
- 页面数据不能长期依赖孤立 mock 列表。

## 公共产品约定

### 移动端基准

当前产品以 `375px` 移动端宽度为基准。页面必须在 375px 下可阅读、可点击、可自然滚动，不出现横向滚动、关键内容遮挡或主按钮不可点击。

### 目的地上下文

Bottle 和 Match 需要承接 Map 中的当前目的地。

推荐路由格式：

```txt
/bottle?dest=<目的地ID>
/match?dest=<目的地ID>
```

`<目的地ID>` 优先使用 `src/pages/Map/data/mapData.ts` 中已有的 region / spot `id`。

### 旅行人格

TripKin 自定义旅行人格使用 `personaId` 作为稳定标识。

稳定标识包括：

- `Cyber-Raider`
- `Zen-Capybara`
- `Budget-Architect`
- `Romantic-Observer`

展示名称包括：

- 赛博特种兵
- 精神卡皮巴拉
- 精算系旅行家
- 人文浪漫主义

Match 页面需要同时展示：

- 16 型 MBTI 字母，例如 `ENFP`、`INTJ`。
- TripKin 自定义旅行人格，例如“精神卡皮巴拉”。

不要把 `mbtiTypeEn` 当作 Match 卡片里的 16 型 MBTI 字母使用。

### 阶段性数据源

当前允许前端本地数据源和 `server/` 内存数据源并存，但页面必须通过 `src/services` 或 `src/store` 访问，不直接引用后端 URL。

继续使用 `mock` 一词时，只表示阶段性数据源或联调层，不表示产品目标仍停留在页面演示。

## 阶段边界

当前阶段不擅自加入：

- 完整登录 / 注册 / 实名系统。
- 数据库。
- 真实 AI 付费接口。
- 真实聊天或好友系统。
- 真实图片上传。
- 真实匹配算法或复杂推荐算法。
- 新 UI 组件库。
- 消息中心。
- 后台管理系统。
- 正式 CI 或完整测试框架。

这些能力不是永久不做，而是必须在对应阶段被明确打开后再做。

## 最小验收标准

- 主链路能从 `/` 进入 MBTI、Map、Bottle、Match 和 Profile。
- MBTI 结果能刷新保留，并在 Profile 展示。
- Map 选择目的地后，Bottle 和 Match 能读取同一目的地。
- Bottle 投瓶后列表变化，且阶段 1 需要同步到用户资产。
- Bottle 收藏、关注、点赞应进入用户资产或明确标注占位。
- Match 申请加入行程应进入申请记录。
- Match 发起同行邀请应进入邀请记录。
- Profile 编辑昵称/签名应持久化。
- 配置 `VITE_API_BASE_URL` 后，已接入 service 的页面应切换到 `server/` API。
- 375px 下没有横向滚动、关键内容遮挡或不可点击主按钮。
