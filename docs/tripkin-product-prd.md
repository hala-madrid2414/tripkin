# TripKin 产品 PRD

本文档是 TripKin 当前 Demo 的产品总说明，只记录已经确认的范围。

## 产品定位

TripKin 是一个移动端 H5 旅行搭子 Demo，用 MBTI / 旅行人格、地图、漂流瓶、搭子匹配和个人身份页串成一条旅行探索体验。

当前目标是做可演示、可截图、可走通的静态前端 Demo，不是生产级完整 App。

## 主链路

```txt
MBTI -> 旅行身份卡 -> Map -> Bottle / Match -> Profile
```

页面路由：

- `/mbti`：旅行 MBTI。
- `/map`：旅行地图。
- `/bottle`：旅行漂流瓶。
- `/match`：搭子 / 行程匹配。
- `/profile`：我的旅行身份。

## 页面职责

### MBTI

作为 Demo 起点，负责完成测试并生成用户的旅行身份。

必须包含：

- 测试入口。
- 答题流程。
- 测试结果。
- 旅行身份卡。
- 进入地图的入口。

### Map

作为 Demo 主枢纽，负责承接 MBTI 后的目的地探索，并分发到 Bottle 和 Match。

必须包含：

- 地图或静态地图兜底。
- 目的地搜索 / 筛选。
- 城市、区域或点位信息。
- 当前目的地上下文。
- 进入漂流瓶和搭子匹配的入口。

### Bottle

作为目的地内容分支，负责展示当前目的地的旅行故事、心愿和漂流瓶互动。

必须包含：

- 当前目的地名称。
- 当前目的地的漂流瓶列表。
- 漂流瓶详情。
- 添加漂流瓶。
- 进入同目的地 Match 的入口。

### Match

作为目的地社交分支，负责展示同目的地搭子和行程。

必须包含：

- 当前目的地名称。
- 搭子 / 行程卡片。
- 筛选。
- 他人身份卡。
- 加入行程。
- MBTI 字母和 TripKin 旅行人格展示。

### Profile

作为用户身份页，负责展示“我的旅行身份”，不是设置页占位。

必须包含：

- 用户旅行身份卡。
- MBTI / TripKin 旅行人格。
- 旅行标签。
- 目的地倾向。
- 漂流瓶、搭子匹配、旅行记忆等摘要入口。

## 弹窗和浮层

已确认需要的弹窗 / 浮层：

- 旅行身份卡。
- 地图筛选。
- 地图搜索建议。
- 漂流瓶详情。
- 添加漂流瓶。
- 他人身份卡。
- 加入行程。

## 公共产品约定

### 移动端基准

当前 Demo 以 `375px` 移动端宽度为基准。

页面必须在 375px 下可阅读、可点击、可自然滚动，不出现横向滚动或关键内容遮挡。

### 目的地上下文

Bottle 和 Match 需要承接 Map 里的当前目的地。

推荐路由格式：

```txt
/bottle?dest=<目的地ID>
/match?dest=<目的地ID>
```

`<目的地ID>` 优先使用 `src/pages/Map/data/mapData.ts` 中已有的 region / spot `id`。

首轮只覆盖地图已有 region / spot，不扩展全量城市库。

### 旅行人格

TripKin 自定义旅行人格使用 `personaId` 作为稳定标识。

已确认展示名称：

- 赛博特种兵。
- 精神卡皮巴拉。
- 精算系旅行家。
- 人文浪漫主义。

Match 页面需要同时展示：

- 16 型 MBTI 字母，例如 `ENFP`、`INTJ`。
- TripKin 自定义旅行人格，例如“精神卡皮巴拉”。

不要把 `mbtiTypeEn` 当作 Match 卡片里的 16 型 MBTI 字母使用。

### 视觉统一

五个页面必须像同一个 TripKin 产品。

需要统一：

- 字体层级。
- 主色和辅助色。
- 页面背景。
- 卡片。
- 按钮。
- 标签。
- 弹层。
- 返回逻辑。
- 信息密度。

`/map` 和 `/match` 可以作为结构和信息密度参考，但不要硬套所有视觉细节。

## 不做范围

当前阶段允许 `server/` 下的最小 mock API 骨架用于后续联调，但不代表真实后端能力完成。当前阶段仍不做：

- 数据库。
- 登录系统。
- 真实 AI。
- 真实聊天或好友系统。
- 真实图片上传。
- 真实匹配算法。
- 新 UI 组件库。

## 验收基准

最小验收标准：

- 主链路能从 `/mbti` 走到 `/map`，再进入 `/bottle` 或 `/match`。
- Bottle 和 Match 能展示从 Map 带入的目的地。
- Profile 首屏能看出用户的旅行身份。
- 375px 下没有横向滚动、关键内容遮挡或不可点击主按钮。
- 五个页面并排看，不像多个独立项目拼在一起。

## 本地验证说明

团队默认使用人工 375px 走查、Chrome DevTools 或普通浏览器检查页面。

`webapp-testing` 只作为少数本地环境已具备该能力的成员的可选辅助方式。AI 或协作者不要主动要求团队成员安装或启用它。只有任务明确点名 `webapp-testing`、Playwright、自动化验证或截图验证，并且当前环境确实可用时，才按 `docs/webapp-testing-guide.md` 执行。

临时脚本、截图和日志仍放在 `.venv/`，不提交到正式目录。
