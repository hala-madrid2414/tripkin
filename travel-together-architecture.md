# TripKin - 技术架构与目录规范初稿

## 0. 命名约定

- 产品展示名：`TripKin`
- Git 仓库名：`tripkin`
- 本地项目目录名：推荐 `tripkin`

## 1. 文档定位

这份文档不是最终交付说明，而是当前比赛节奏下的技术架构底稿，用来指导：

- 仓库怎么搭
- 页面怎么拆
- 代码往哪里放
- 5 个人如何低干扰协作

## 2. 当前技术方向

### 已确定

- 前端框架：`React + Vite + TypeScript`
- 路由：`React Router`
- 状态管理：`Zustand`
- 请求库：`Axios`
- 样式方案：`CSS Modules + Less`
- 包管理器：`npm`
- 路径别名：`@ -> src`
- 根路径处理：`/` 重定向到 `/mbti`

### 暂未最终确定

- 组件库是否接入、接哪个

### 当前明确不做

- monorepo
- 当前阶段的后端仓库
- 数据库
- 登录体系
- 重 CI 流程

---

## 3. 架构目标

当前架构的目标不是一步到位，而是：

1. 先支持按页面分工
2. 先让静态页面、跳转链路和截图能力成立
3. 后续再逐步补模块逻辑、mock、接口、AI 能力
4. 保证多人协作时尽量少互相干扰

---

## 4. 总体架构思路

当前采取“以前端壳子为中心”的结构。

原则：

- 以页面为主组织开发
- 以路由承接主流程
- 以模块拆页面内部大块
- 以组件承接跨页面复用
- 以轻工程化保障协作质量

也就是说，当前不是先围绕数据模型或数据库设计，而是先围绕：

- 页面结构
- 主流程
- 协作边界

来组织工程。

## 6. 各目录职责

### `src/pages/`

放带路由的大页面。  
这些页面属于用户主流程中的独立一步。

### `src/modules/`

放页面内部的大功能块。  
它们不单独挂路由，但值得独立拆分。

### `src/components/`

放跨页面复用组件。  
比如卡片、标签、弹窗容器、通用头部、底部操作栏。

### `src/router/`

放路由统一配置。

### `src/store/`

放 Zustand 的全局状态。

### `src/services/`

放请求层和后续数据访问入口。  
当前可以先留空，不急着做完整接口。

### `src/styles/`

放全局样式、重置样式、变量。

### `src/utils/`

只放纯工具函数。  
不要把业务逻辑混进 `utils`。

### `src/types/`

放公共 TS 类型。  
页面私有类型可以先放页面内部。

### `src/assets/`

放图片、图标、插画、静态资源。

---

## 7. 页面、模块、组件的边界

### 带路由页面

用于承接主流程中的独立一步。  
用户进入后，算进入新的阶段。

### 不带路由模块

用于某个页面内部的信息切换、局部展开、结构变化。  
不单独作为流程中的新一步。

### 弹窗

用于轻量展示、说明、确认、分享。  
只在当前页面上临时出现，不承接完整主流程。

### 可复用组件

用于跨页面复用的统一结构。  
不代表独立流程步骤。

## 12. 状态管理边界

当前使用 `Zustand`，但只承接真正跨页面共享的状态。

适合进 store 的：

- 用户 MBTI 结果
- 当前身份/卡片相关状态
- 地图筛选状态
- 匹配相关状态

不建议进 store 的：

- 某个按钮 hover
- 某个弹窗局部开关
- 单页内部的一次性小交互

原则：

**能用本地 state 就本地 state，必须跨页面共享再进 Zustand。**

---

## 13. 请求与数据层

当前请求库统一为 `Axios`。

规则：

- 页面内不直接散写请求
- 后续统一通过 `src/services/` 收口
- 当前阶段可先不细化完整接口文件

后续常见结构参考：

```txt
services/
  request.ts
  mbti.ts
  map.ts
  bottle.ts
  match.ts
```

当前 mock 策略：

- 不单独先搭大 mock 体系
- 局部展示数据可先局部使用
- 出现复用后再抽离

## 14. 样式体系

当前统一方案：

- 页面/组件：`CSS Modules + Less`
- 全局：`src/styles/`

推荐全局样式文件：

```txt
styles/
  reset.less
  global.less
  variables.less
```

原则：

- 默认不用普通 `.less`
- 不随便新增全局类
- 不混 Tailwind

### 移动端开发基准

当前统一口径：

- 设计与开发统一按 `375px` 宽基准进行
- 页面高度不锁死，不按 `375 * 750` 作为固定画布开发
- 高度按内容自然延展
- 当前阶段样式可以先以 `px` 为主
- 但布局结构从第一天开始就按可伸缩方式处理

推荐做法：

- 主容器 `width: 100%`
- 主容器可设置 `max-width`
- 页面使用 `min-height: 100dvh`
- 布局优先使用 `flex` / `grid`
- 不把整页结构写死成固定宽高

适合直接使用 `px` 的内容：

- 字号
- 间距
- 圆角
- 图标尺寸
- 卡片内边距
- 按钮高度

需要从一开始就考虑伸缩的内容：

- 整页容器宽度
- 地图区域
- 双列或横向布局
- 底部操作区
- 弹窗边距
- 长文案换行

一句话规则：

**按 `375px` 宽开发，样式可先写 `px`，但布局结构必须从一开始就是移动端可伸缩写法，不做“先固定后整体转响应式”的方案。**

## 15. 工程化边界

当前只保留轻量工程化：

- `ESLint`
- `Prettier`
- `Husky`
- `lint-staged`

目的：

- 减少低级错误
- 统一代码格式
- 不增加协作阻力

当前不优先引入：

- `commitlint`
- `stylelint`
- 提交前跑测试
- 复杂 CI 卡点

---

## 16. AI 协作建议

当前阶段不建议一开始就写很多 skill。

更有效的做法是先补齐少量硬规则文档：

- `README.md`
- `docs/architecture.md`
- `docs/collaboration.md`

AI 协作核心不是“配置越多越高级”，而是：

- 目录清楚
- 规则短且一致
- 页面边界明确
- 不让 agent 到处乱放代码

---

## 17. 协作原则

当前建议：

1. 按页面分工
2. 尽量不要跨着改别人页面主结构
3. 复用能力优先抽到 `components/`
4. 页面内部大块抽到 `modules/`
5. 所有人遵守统一命名和样式规则

## 19. 当前一句话总结

先用 `React + Vite + TS` 搭一个以页面为中心、支持多人并行、轻工程化的移动端 H5 前端壳子；先保证主流程页面、路由跳转和协作边界清楚，再逐步补模块、交互、数据、接口和后端扩展。
